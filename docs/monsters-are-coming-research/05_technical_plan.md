# 05. 技术方案

## 推荐引擎

### Unity

适合快速做 2D/2.5D 自动战斗原型。优势是生态成熟、寻路和 UI 快速、移动端/PC 都方便。SteamDB 显示原作使用 Unity Engine，这也说明该类型的移动基地、自动战斗和大量实体压力在 Unity 路线上可行。大量实体时建议使用对象池和空间分区；如果后期单位量很高，可考虑 DOTS，但 MVP 不必一开始引入。

### Godot

适合小团队和轻量原型。GDScript 迭代快，2D 工具链简单。大量敌人时需要更早关注节点数量和物理查询成本。

### Unreal

适合做高表现 3D 版本，但对这个题材的 MVP 成本偏高。

建议：若目标是 4-8 周可玩原型，优先 Unity 或 Godot。

## 核心模块

| 模块 | 职责 |
| --- | --- |
| GameSession | 单局状态、时间轴、暂停、胜负 |
| CaravanCore | 移动城镇生命、路径推进、建造网格 |
| ResourceSystem | 资源点、掉落、采集、库存 |
| BuildSystem | 建筑放置、成本、升级、拆除 |
| CombatSystem | 目标选择、伤害、投射物、状态效果 |
| SpawnDirector | 波次预算、刷怪方向、精英/Boss |
| UpgradeSystem | 升级池、稀有度、三选一 |
| MapGenerator | 路径、资源簇、事件点、障碍 |
| MetaProgression | 解锁、收藏、难度、存档 |
| Telemetry | 单局数据、热力图、平衡分析 |

## 数据驱动

建议所有内容用表驱动：

- `buildings.json`：成本、生命、射程、伤害、升级树。
- `enemies.json`：生命、速度、伤害、AI 类型、预算点数。
- `waves.json`：时间、预算、敌人池、特殊事件。
- `upgrades.json`：权重、稀有度、效果脚本 ID。
- `resources.json`：采集时间、产量、地图权重。

## 性能策略

- 敌人和投射物使用对象池。
- 战斗目标选择使用空间哈希或网格分桶，避免每帧全量遍历。
- 远离相机的资源点降低刷新频率。
- 敌人 AI 分层更新：近距离每帧，远距离每 0.2-0.5 秒。
- 投射物命中用简化圆形碰撞，必要时只让关键弹道使用物理系统。
- 数值变化和伤害飘字限流，防止 UI 压力。

## 地图与移动基地实现

推荐方案：

1. 地图世界静止，城镇沿路径移动。
2. 建筑作为城镇子对象或跟随核心位置偏移。
3. 建造格子使用本地坐标存储，例如 `gridOffset = (x, y)`。
4. 世界坐标由 `corePosition + rotate(gridOffset)` 计算。
5. 敌人寻路目标使用核心或建筑世界坐标。

注意点：

- 如果城镇旋转，建筑射界和格子对齐会复杂；MVP 建议城镇不旋转，只沿路线平移。
- 如果地图很大，可以做环形加载区块，城镇经过后回收旧区块。

## AI 设计

敌人状态机：

- Spawn：出生预警。
- Approach：接近目标。
- Attack：进入射程后攻击。
- Retarget：目标死亡或路径堵塞时重选。
- Stagger：被击退/眩晕。
- Die：掉落资源，回收到对象池。

目标评分：

```text
score = distanceWeight + threatWeight + tauntWeight + objectiveWeight
```

MVP 可先实现“最近有效目标”，后续再加评分。

## 存档结构

- 解锁角色列表。
- 解锁建筑蓝图。
- 难度通关记录。
- 收藏品/图鉴进度。
- 设置项。

单局中断存档可以后置；MVP 只需要局外进度。

## 调试工具

- 波次预算实时显示。
- 敌人目标线开关。
- 建筑 DPS 统计。
- 资源生成热力图。
- 一键跳到第 N 分钟。
- 升级池强制抽取。

## 自动化测试建议

- 数据表加载测试：所有 ID 引用有效。
- 波次预算测试：每个时间点可生成敌人。
- 建筑成本测试：资源扣除不为负。
- 伤害公式测试：护甲、状态、暴击可预测。
- 存档迁移测试：新增字段有默认值。
